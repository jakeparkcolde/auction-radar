# SPEC-UI-001 구현 계획

> Traceability: SPEC-UI-001 (spec.md) · 기획서 v0.2 §4.2 F5, §11, §12 (M3)

## 1. 마일스톤 (우선순위 기반)

### Primary Goal — 읽기 전용 데이터 레이어 & 서버

1. `apps/dashboard` 골격: 로컬 HTTP 서버, `127.0.0.1` 바인딩 고정 (REQ-002)
2. 읽기 전용 쿼리 레이어: items/events/schedules/matches/sync_runs 조회 전용 (REQ-001) — 스토어 드라이버의 read 경로만 사용
3. JSON 데이터 엔드포인트 (localhost 전용) 또는 정적 데이터 임베드 방식 결정

### Secondary Goal — 단일 HTML 화면

1. 물건 목록 뷰: 감정가·최저가·유찰·D-day, 억/만 환산 유틸 재사용 (REQ-004)
2. 이벤트 이력 타임라인 (REQ-004)
3. 필터: 워치리스트 / 이벤트 타입 / 기간 (REQ-007)
4. 상태 배너: 마지막 sync 실패·차단 경고 (REQ-008)
5. 면책 고지 고정 + 법원 원문 링크 (REQ-006)
6. 에셋 전량 인라인 — 외부 요청 0건 빌드 검증 (REQ-003)

### Final Goal — enrich 표시 & 마감

1. 할인율·표본·신뢰도 표시 (강조 규칙 SPEC-ENRICH-001 계약 준수) (REQ-005)
2. D-7 임박 섹션 (REQ-009, optional)
3. M3 DoD 통합 확인: 대시보드가 알림과 동일 events 데이터를 표시

## 2. 기술 스택

| 구성요소 | 선택 | 비고 |
|---|---|---|
| 서버 | node:http (내장) | 단일 HTML 서빙 + JSON 엔드포인트. 프레임워크 불요 |
| 프론트 | Vanilla TS + 경량 템플릿 (또는 preact >= 10 단일 번들) | 결정 노트: v1.x 단일 HTML 원칙상 React/Next.js 과함. 번들 후 인라인 가능한 초경량만 허용. Next.js 전환은 별도 SPEC |
| 번들러 | esbuild >= 0.23 | 단일 파일 인라인 번들(CSS/JS 포함) 생성에 최적 |
| 스토어 접근 | packages/store 읽기 경로 | prepared statements, 쓰기 API 미참조를 lint 규칙/리뷰로 강제 |
| 테스트 | vitest + happy-dom (렌더 단위), 서버 통합 테스트 | 외부 요청 0건 검증 포함 |

주: 정확한 안정 버전은 /moai:2-run 단계에서 최종 확인.

## 3. 다른 SPEC과의 의존 관계

- **선행**: SPEC-COLLECTOR-001 — 스토어 스키마·events 스트림 (하드 의존)
- **소프트 의존**: SPEC-ENRICH-001 — 할인율 표시. **부재 시에도 대시보드는 완전 동작해야 함** (할인율 컬럼 미표시)
- **공유 유틸**: 가격 억/만 환산·D-day 계산은 SPEC-ALERT-001 렌더러 유틸 재사용 (표기 불일치 방지)
- **병행 가능**: SPEC-ENRICH-001과 병행 개발 가능 (M3 내 병렬 트랙)

## 4. 리스크 분석 및 대응

| 리스크 | 영향 | 대응 |
|---|---|---|
| 실수로 외부 노출 (0.0.0.0 바인딩) | 상 | 바인딩 주소 하드코딩 + 통합 테스트로 외부 인터페이스 접근 불가 검증 |
| 쓰기 경로 유입 (기능 추가 시) | 중 | 읽기 전용 쿼리 레이어 분리 + 코드 리뷰 게이트, 편집 기능은 백로그로 명시 격리 |
| CDN 에셋 의존 유입 | 중 | 빌드 후 외부 URL 존재 여부 정적 검사 (esbuild 산출물 grep) |
| 알림과 대시보드 표기 불일치 | 중 | 렌더링 유틸 공유(단일 소스), 스냅샷 테스트 교차 검증 |
| 단일 HTML 성능 저하 (물건 수천 건) | 저 | 페이지네이션/가상 스크롤은 필요 시 도입, 초기엔 최근 90일 기본 필터 |

## 5. 테스트 전략

- 서버 통합 테스트: 127.0.0.1 바인딩 확인, 읽기 전용(쓰기 시도 시 코드 경로 부재) 확인
- 렌더 단위 테스트: 가격 환산·D-day·신뢰도 강조 규칙 (SPEC-ENRICH-001 계약 fixture 재사용)
- 정적 검사: 번들 산출물에 외부 URL(http/https CDN) 0건
- enrich 부재 시나리오: rt_trades 빈 DB로 전체 화면 렌더 성공
- 커버리지 목표: 데이터 레이어·렌더 유틸 85%+ (마크업 레이어는 스냅샷 테스트로 대체 가능, 사유 명기)
